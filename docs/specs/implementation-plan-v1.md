# Trusted Installer Corpus — v1 Implementation Plan

**Status:** Spec approved (per handoff sign-off) — this plan implements it as written, no re-litigation of decisions already made.
**Source of truth for architecture/schema/rationale:** `docs/specs/design-v1.md`. This document only breaks that spec into executable tasks; it does not restate the "why."
**Suggested location for this file in the repo:** `docs/plans/v1-implementation-plan.md`

## Repo layout for v1

This is a new decision (open item from the handoff — "choose optimal locations yourself"). Rationale: keep the spec's flat top-level scripts (matches the pipeline diagram exactly, easy to `python crawler.py` / `python enrich.py` etc.), but pull shared logic and the seed-list scraper implementations into their own directories so they're independently testable.

```
trusted-installer-corpus/
├── docs/
│   ├── specs/
│   │   └── design-v1.md
│   └── plans/
│       └── v1-implementation-plan.md
├── seed_sources/
│   ├── manual_seed_list.json          # hand-picked ~50-100, source of truth for "manual"
│   ├── scrape_awesome_cli.py          # scraper, writes candidate entries
│   ├── scrape_webinstall_dev.py       # scraper, writes candidate entries
│   └── build_seed_list.py             # merges manual + scraped -> seed_list.json (dedup, schema-validate)
├── lib/
│   ├── db.py                          # SQLite connection + schema init (CREATE TABLE IF NOT EXISTS)
│   ├── github_auth.py                 # `gh auth token` shell-out, raises clear error if unavailable
│   ├── url_utils.py                   # domain/path parsing — pure functions
│   └── rate_limit.py                  # shared backoff/retry helpers for crawler + enrich
├── crawler.py
├── enrich.py
├── aggregate.py
├── export.py
├── seed_list.json                     # generated by build_seed_list.py (not hand-edited directly)
├── corpus.db                          # generated, gitignored
├── corpus.json                        # generated output, the deliverable artifact
├── tests/
│   ├── test_url_utils.py
│   ├── test_aggregate.py
│   └── test_truncation.py
├── requirements.txt
└── .gitignore                         # corpus.db, __pycache__/, .pytest_cache/
```

`seed_list.json` is treated as a build artifact of `build_seed_list.py`, not something edited by hand — this keeps "manual" entries auditable in `manual_seed_list.json` while still letting the combined list be regenerated whenever the scrapers pick up new candidates.

## Phase 0 — Scaffolding

**T0.1 — Repo skeleton**
- Create directory structure above, `requirements.txt` (`requests`, `PyGithub`, `pytest`), `.gitignore`.
- Acceptance: `pip install -r requirements.txt` succeeds; empty package imports don't error.

**T0.2 — `lib/github_auth.py`**
- Shell out to `gh auth token`, return the token string.
- Fail fast with a clear message if `gh` isn't installed or isn't authenticated (don't silently fall back to unauthenticated requests — spec's rate limits assume authenticated calls).
- Acceptance: unit test mocks `subprocess.run`; one live check documented in README ("run `gh auth status` first").

**T0.3 — `lib/db.py`**
- Implement the schema from the spec verbatim:
  `seed_urls`, `raw_hits`, `repos`, `corpus`.
- `init_db()` is idempotent (`CREATE TABLE IF NOT EXISTS`), safe to call from every stage script.
- Acceptance: running `init_db()` twice against the same file is a no-op on the second call; schema matches spec field-for-field.

## Phase 1 — Seed list

**T1.1 — `lib/url_utils.py`: domain/path splitting**
- Pure function `split_url(url) -> (domain, path)`.
- Acceptance: unit tests cover bare domains, paths with query strings, trailing slashes.

**T1.2 — Manual seed list**
- Populate `seed_sources/manual_seed_list.json` with ~50-100 hand-picked, well-known installer URLs (the ones named in the spec's integration test — `bun.sh/install`, `sh.rustup.rs`, `get.docker.com` — must be in this set).
- Each entry: `{"url": ..., "domain": ..., "source": "manual"}`.
- Acceptance: valid JSON, no duplicate URLs, every entry has a non-empty domain.

**T1.3 — `seed_sources/scrape_awesome_cli.py`**
- Fetch and parse the awesome-cli list for installer-style URLs (`curl | bash`, `install.sh` links, etc.).
- Output candidates tagged `"source": "scraped:awesome-cli"`.
- Acceptance: run against live source, produces >0 candidates, each passes `split_url` without raising.

**T1.4 — `seed_sources/scrape_webinstall_dev.py`**
- Same shape as T1.3, scraping webinstall.dev.
- Acceptance: same as T1.3.

**T1.5 — `seed_sources/build_seed_list.py`**
- Merge manual + all scraped sources, dedupe by exact URL, write combined `seed_list.json` matching the spec's schema.
- Acceptance: output validates against schema; no duplicate URLs; count is logged (manual vs. scraped breakdown) so a human can sanity-check before the crawl burns rate-limit budget.

---

## Phase 2 — Crawler (`crawler.py`)

**T2.1 — Query construction**
- Build the exact-literal-match query per seed URL: `"<url>" in:file`.
- No regex — confirmed unsupported by REST Code Search (per spec).
- Acceptance: unit test on query string construction (escaping quotes in URLs, if any).

**T2.2 — Rate limiting**
- Respect 10 req/min authenticated cap. Sleep between calls; read `X-RateLimit-Remaining` from response headers and back off proactively, not just on 403.
- Acceptance: instrumented dry run against a handful of seeds shows no 403s from rate limiting under normal operation.

**T2.3 — Pagination + truncation flag**
- Page through results (100/page, up to 10 pages = 1000 cap). If `total_count` hits the cap, set `truncated = true` for that seed URL.
- Acceptance: unit test on the truncation-flag logic using a mocked `total_count` at/under/over the cap.

**T2.4 — Resumability**
- Before querying a seed URL, check if it already has rows in `raw_hits` for the current run; skip if so.
- Acceptance: killing the crawler mid-run and re-running doesn't re-query already-crawled seed URLs (verify via call count / mock).

**T2.5 — Error handling**
- Network error / 5xx: retry 3x exponential backoff, then log and skip (don't kill the run).
- 403 / secondary rate limit: back off and retry (separate path from T2.2's proactive throttling).
- Acceptance: unit tests simulate a 5xx and a 403 and confirm the crawler continues to the next seed URL rather than crashing.

---

## Phase 3 — Enrich (`enrich.py`)

**T3.1 — Distinct repo extraction**
- Pull the distinct set of `repo_full_name` values from `raw_hits`.
- Acceptance: unit test with duplicate repo names across multiple seed URLs collapses to one entry.

**T3.2 — Repo metadata fetch + cache**
- For each distinct repo, `GET /repos/{owner}/{repo}` for stars/owner; write to `repos` table.
- Skip repos already present in `repos` (cache hit) — this is the point of caching, since one repo can reference multiple seed URLs.
- Acceptance: a repo referenced by 3 different seed URLs triggers exactly one metadata fetch.

---

## Phase 4 — Aggregate (`aggregate.py`)

**T4.1 — Join + counts**
- Join `raw_hits` + `repos`, compute `occurrences`, `unique_repositories`, `unique_owners` per seed URL.
- Acceptance: unit test with a small fixture dataset checks exact counts.

**T4.2 — Star statistics**
- Pure functions for `median_repo_stars` and `max_repo_stars`.
- Acceptance: unit tests including edge cases (0 repos, 1 repo, even-count median).

**T4.3 — Timestamps + truncated carryover**
- `first_seen` / `last_seen` per seed URL; carry the `truncated` flag from the crawl stage into the `corpus` row.
- Acceptance: re-running aggregate after a later crawl updates `last_seen` without clobbering `first_seen`.

---

## Phase 5 — Export (`export.py`)

**T5.1 — JSON export**
- Dump the `corpus` table to `corpus.json` matching the target dataset schema.
- Acceptance: output validates against the schema; spot-check a few entries against the SQLite source rows.

---

## Phase 6 — Testing

**T6.1 — Unit test suite**
- Covers: `url_utils.split_url`, aggregation math (median/max), truncation-flag logic. All pure functions, no live API calls, per spec.
- Acceptance: `pytest` green, no network access required to pass.

**T6.2 — Integration run (manual)**
- Run the full pipeline (crawler → enrich → aggregate → export) against 3-5 known seed URLs: `bun.sh/install`, `sh.rustup.rs`, `get.docker.com` (plus 1-2 more).
- Sanity-check the resulting `corpus.json` numbers against what a human would expect (these are extremely widely-referenced installers — occurrence counts should be non-trivial, not zero or obviously wrong).
- Acceptance: numbers look sane to a human reviewer; no crashes across all four stages.

---

## Phase 7 — Full run

**T7.1 — Full seed list run**
- Only after T6.2 passes: run the pipeline against the full ~500-entry `seed_list.json`.
- Given the 10 req/min Code Search cap, budget the crawl time accordingly (~500 seeds → expect the crawl stage alone to take on the order of an hour; confirm actual seed count once T1.5 is done and adjust expectations).
- Acceptance: `corpus.json` produced for the full list; truncation flags reviewed for any seed URL that hit the 1000-result cap.

---

## Suggested execution order

Phase 0 → Phase 1 → Phase 2 → Phase 3 → Phase 4 → Phase 5 → Phase 6 (T6.1 can actually run in parallel with Phase 2-5 development, since it's pure-function tests with no dependency on live stages) → Phase 6 T6.2 → Phase 7.

## Definition of done for v1

Matches the spec's own success criteria: the pipeline produces sane, evidence-backed occurrence/repo/owner/star numbers for known-legitimate installers, run manually against the full seed list, with no auto-discovery, no scheduling, and no combined scoring formula. Nothing in this plan should introduce those out-of-scope pieces — if a task seems to need them, flag it rather than quietly expanding scope.
